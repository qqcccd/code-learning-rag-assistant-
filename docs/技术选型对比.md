# 《代码学习智能审查助⼿》技术选型对⽐⽂档
## ⼀、项⽬背景
本项⽬是基于RAG与⼤模型的编程答疑系统，核⼼⾯向编程学习者，提供代码分析、智能答疑能⼒，技术选型核⼼围绕「开
发效率⾼、适配AI场景、复试可讲性强、轻量易部署」四个核⼼原则。
## ⼆、核⼼技术选型对⽐
### 1. 后端框架选型：FastAPI vs Flask
| 对⽐维度 | FastAPI | Flask | 最终选型 |
|----------|---------|-------|----------|
| 性能     | 基于异步Starlette框架，并发性能优异，适配AI接⼝⾼并发场景 | 同步框架，并发性能弱，⾼并发场
景需额外改造 | FastAPI |
| 开发效率 | 原⽣⽀持类型提示，⾃动参数校验，⾃动⽣成OpenAPI接⼝⽂档，减少重复代码 | ⼿动编写参数校验、接
⼝⽂档，开发效率低 | FastAPI |
| ⽣态适配 | 与LangChain、⼤模型⼯具链原⽣适配，AI应⽤开发⽣态完善 | 通⽤Web框架，AI场景适配需额外开发 
| FastAPI |
| 学习成本 | 基于Python标准，上⼿简单，⽂档完善 | 上⼿简单，但⾼阶功能需额外学习 | 持平 |
选型理由：FastAPI的异步性能、⾃动接⼝⽂档能⼒，更适配AI应⽤的开发场景，同时在复试中能体现对现代Web框架的
理解，拉开与Flask的同质化差距。
### 2. 向量数据库选型：ChromaDB vs FAISS
| 对⽐维度 | ChromaDB | FAISS | 最终选型 |
|----------|----------|-------|----------|
| 部署难度 | 轻量本地持久化部署，单⽂件存储，⽆需额外服务 | 仅内存索引，持久化需额外开发，部署复杂 | 
ChromaDB |
| 功能完整性 | 原⽣⽀持元数据过滤、⽂档增删改查，适配RAG全流程 | 仅核⼼向量检索能⼒，周边功能需⾃⾏开发 | 
ChromaDB |
| 性能 | 轻量场景下性能优异，满⾜万级以内⽂档检索需求 | 海量数据场景性能更强，本项⽬⽤不到 | 持平 |
| 开发效率 | Python SDK完善，3⾏代码即可完成向量库创建 | 需⾃⾏封装索引、持久化逻辑，开发成本⾼ | 
ChromaDB |
选型理由：ChromaDB轻量易部署，原⽣⽀持持久化，完全满⾜本项⽬的知识库规模需求，开发效率⾼，复试中能完整讲解
RAG的向量存储全流程。
### 3. ⼤模型API选型：DeepSeek API vs 通义千问/⽂⼼⼀⾔
| 对⽐维度 | DeepSeek API | 通义千问/⽂⼼⼀⾔ | 最终选型 |
|----------|--------------|--------------------|----------|
| 代码能⼒ | 原⽣针对代码场景优化，代码理解、纠错、解释能⼒优异 | 通⽤场景能⼒强，代码垂直场景能⼒弱于
DeepSeek | DeepSeek |
| 调⽤成本 | 输⼊成本极低，百万token仅需1元，全项⽬开发成本不超过5元 | 调⽤成本更⾼，同等token量成本是
DeepSeek的3-5倍 | DeepSeek |
| ⻓上下⽂⽀持 | 原⽣⽀持128K⻓上下⽂，适配⻓代码⽂件分析场景 | ⻓上下⽂需更⾼版本接⼝，成本更⾼ | 
DeepSeek |
| 开发适配 | API完全兼容OpenAI格式，SDK简单，上⼿快 | 接⼝格式有差异，需额外适配 | DeepSeek |
选型理由：DeepSeek在代码垂直场景的能⼒更强，调⽤成本更低，完全适配本项⽬的编程答疑场景，复试中能体现对垂直
领域⼤模型选型的思考。
